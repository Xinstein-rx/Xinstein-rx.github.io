为什么要学习git规范

  1. 自动生成CHANGELOG.md
  2. 提供清晰的历史记录
  3. 可以忽略不重要的提交，如仅仅是代码格式的改变

commit message的格式

提交信息的任何一行都不能超过100个字符!

一条提交信息包括一个header, 一个 body 和一个 footer，每部分之间用空行分隔。

Message header

Message header是一个单行，包含对change的简洁描述，包括一个type、一个optional scope和一个subject。

允许的 type

允许的 scope

scope可以是任何指定提交修改的地方，如 $location, $browser, $compile, $rootScope, ngHref, ngClick, ngView…

如果没有合适的scope，可以使用 *

subject内容

subject是一个对change简短的描述

  1. 使用祈使句，现在时态：“change” not “changed” nor “changes”
  2. 首字母不要大写
  3. 结尾不要有 .

Breaking changes

不兼容修改指的是本次提交修改了不兼容之前版本的API或者环境变量

所有不兼容修改都必须在页脚中作为中断更改块提到，以BREAKING CHANGE:开头，后跟一个空格或者两个换行符，其余的信息就是对此次修改的描述，修改的理由和修改注释

执行 npm run release 指令实际执行了五个动作

  1. 修改 package.json 中的版本号
  2. 修改 package-lock.json 中的版本号
  3. 生成 CHANGELOG.md 文件
  4. 提交 package.json、package-lock.json、CHANGELOG.md 文件
  5. 给这次提交记录打上tag
