1.5 组合总和Ⅲ

题目链接：https://leetcode.cn/problems/combination-sum-ii/

这个题目每个字符只能用一次了，但是有个很棘手的问题，就是题目给的数据，集合（数组candidates）有重复元素，但还不能有重复的组合, 需要对这些重复的集合进行去重。一开始我想到的办法就是把所有符合条件的集合先放进来，到dfs完成之后再去重，但是这样很容易就超时了。

所以去重的工作需要在dfs的时候就得完成。回溯算法一般来说都是依赖递归，递归的话就会有一颗递归树，那去重是在某一层去重还是在某一条路径上去重是不一样的。这个题的话就是在相同一层去重的，因为路径上相同并不影响这个集合，但是同一层相同就会出现相同的集合。

那现在的问题变成了如何做到在某一层去重？

自然想到的，定义一个标记数组used来标记一下，这个数字这一层有没有用到过，那什么时候需要判断呢，就是这个数字不是第一次出现了，既然第二次出现，可以对原数组排一个序，如果nums[i] == nums[i - 1]，也就是说当前数字等一前一个数字，那就不是第一次出现了， 需要注意重复的问题。当然也可以不用排序，但是不排序的话急需要线性的去遍历是不是出现过这个数字，这带来的时间消耗就很大了。

如果出现nums[i] == nums[i - 1] && used[i - 1] == false，就说明前一个数字在本层已经用过了，这个看起来还挺玄乎的，为啥used[i - 1] == false，反而判断成本层已经用过了，而按照逻辑不是true的时候，才算是用过吗？

我是这样考虑的，这个used数组说是标记数组，看用没用过，但其实是用来区分是树枝还是树层的，站在树的同一层考虑，只要是出现了nums[i] == nums[i - 1]，就不能再从这个i下标dfs下去了，因为前面已经有重复的情况了，和used没有什么关系，但是站在同一条路径的角度，如果出现了nums[i] == nums[i - 1]，那其实是没有关系的，还可以继续往下遍历，同样的出现nums[i] == nums[i - 1]，那就需要有一种手段区分这两种情况，不能因为重复了就不往下dfs了，毕竟同一路径上的重复还是可以继续往下dfs的。so，used数组就是遇到相等的情况区分同一层还是同一路径的，是同一路径的话，那当然used[i - 1] = true。

2 求分割

2.1 分割回文串

题目链接： https://leetcode.cn/problems/palindrome-partitioning/

这个题我记得是卡在了如何截取字串上面，但其实还是当时对递归递归理解的不深，既然都dfs了，那必然是每一个位置都会当成字串的起点，肯定会遍历到所有的字串，只需要substr选取这些字符串就好了。

其实分割回文串这里和上面的组合方法是一样的，只不过自己陷入了一个思维定式，就是每一层dfs，都是要选择出一个字符，第一次选第一个，第二次选第二个，都是单独选的，但其实，这样是不符合切割的条件。

正确的想法应该是，每层dfs，第一次切割第一个字符，第二次第一个字符连同第二个字符一起被切割，这样才是连续的，才能把每个字串都顺利的找出来。

这个题是要子集不能有重复，但是元素是有重复的，还是之前说的那个去重问题，排一下序，这个很重要，然后used数组区分是同一层重了，还是同一条路径重了。

3.3 递增子序列

题目链接：https://leetcode.cn/problems/increasing-subsequences/

这个题和子集问题很像，但是要求子集是递增的，但是数组里给出了重复元素，说明有可能有重复的子集，还是需要对子集进行去重。但是这里的去重就不能再用排序+used数组了，因为是原序列上的递增，排序之后序列顺序就变了。所以要采取另外的方法。

这个方法就是针对于每一层定义一个标记数组，以前的方法是一整颗树有一个标记数组，这里的是每一层有一个标记数组，也可以是个set。如果这一层的标记数组标记过这个元素已经使用过了，即used[nums[i] + 100] == true，或者是set.count()不是0，那就说明这一层这个元素重复了，元素的值+100的原因是因为取值范围是[-100, 100]。

5 棋盘问题

5.1 N皇后问题

题目链接：https://leetcode.cn/problems/n-queens/

这道题没啥太难的，就是一开始没想到如何把棋盘保存下来，看来只会算法也不行的，还得有好想法知道是如何输入输出的

我一开始想的是，先保存每一层皇后的位置，最后到叶子节点的时候，再把棋盘转换成string的形式，但其实多此一举了，完全可以先定义好棋盘，每次dfs下一层，就把那个位置设置成‘Q’，回溯完了再置为‘.’

还有一个很关键的地方就是如何判断皇后是合法的，这个讲过好多好多遍了，大一的时候一个老师就讲过了。可以三个循环分别判断列，左上，右上，是不是都满足条件。

5.2 解数独

题目链接：https://leetcode.cn/problems/sudoku-solver/

如何判断小棋盘是不是合法还是很好判断的，我觉得这个题对我来说，难就难在如何去循环dfs上，这个题需要一个二维循环的dfs，以前遇到的dfs都是一层循环就能搞定，这次二层的循环，然后还需要一层从0-9的循环。

还以为和n皇后一样，一维遍历就可以，用定义bool的递归函数形式，来结束递归，即在找到一个解之后就return true，这种方法非常适合只需要找一个解的情况，只要找到了就返回不找了。

本来这个题还不信邪，想用一维的方式来做，但是如果是用一维，倒是也能做，就是超时了，如果找到就return true的方式，一维的dfs，会导致只能填上这一行的第一个空，剩下的不会填上，所以还是需要二维的遍历，这样就算是这个位置找到了一个合适的数字，返回了true，根据二维的循环，还能继续填下一个空。

6 其他问题

6.1 重新安排行程

题目链接：https://leetcode.cn/problems/reconstruct-itinerary/

这个题考的是欧拉回路，经过每条边，且只经过一次

在回溯法的过程中体会到一个事情，就是每次递归可以不用把下一步传进去，直接在循环逻辑里进行下一步

这个题很奇怪，dfs里加引用和不加引用出现的结果是不一样的，抽时间把c++里的引用好好看看

用来记录两个城市之间的关系的数据结构也需要注意一下， 我一开始把城市转成了对应的数字，当作节点的编号，然后用二维的vector来记录关系，但是好像这样会很麻烦，需要记录string和int编号的双向对应，很容易就把自己搞迷糊了，其实可以直接搞一个string 的关系对应，即unordered_map<string, map<string, int> >mp，因为这个题要求是按照字典序进行遍历，返回最小的遍历序列，所以第二维应该是有序的，明明有这么好用的容器，我却还自己傻傻的排序，太傻啦！！

因为找欧拉回路的时候需要把遍历过的路径从图中去掉，但是真的做去掉这个操作，一是时间花费代价太大，二是真的删掉边之后会改变之前的字典序，要想回溯的话，就会出错，所以可以通过记录路径个数的方法，每次dfs之前，先看看与这个节点连的边是不是0，不是0再dfs，把记录边个数的值减去1，等回溯回来的时候再加上1.
